# План доработок поведения бота

## Контекст

Список правок поведения Instagram-бота турагентства «Сандита». Почти всё меняется в системном промпте (`src/ai/prompts.py`), плюс немного логики в `src/main.py` (флаг приветствия, захват телефона из эскалации). Цель: бот не здоровается дважды, говорит о себе в единственном числе («я» — чтобы было понятно, что это бот), корректно передаёт клиента менеджерам Наталье и Татьяне с номером телефона, спрашивает имя при первом контакте и даёт точный ответ, когда подходящих туров нет.

Уточнено с заказчиком:
- номер менеджеров — **+375 29 152 37 28** (имена Наталья и Татьяна);
- приглашение оставить номер — **по контексту** (не в каждом сообщении).

Вопрос про размещение на серверном компьютере — информационный, кода не требует: приложение уже Dockerized (`Dockerfile`, `docker-compose.yml`), для self-host нужны публичный HTTPS-эндпоинт (Meta шлёт webhook только на https), белый IP/домен (есть поддержка DuckDNS), ~2 ГБ RAM и аптайм 24/7.

## Изменения

### 1. `src/ai/prompts.py` — основное

**a) Единственное число «я» вместо «мы»** (строка 56 `_BASE_RULES`, строка 87 `_SECURITY_RULES`)

Новая строка про самоидентификацию:

> О себе говори в единственном числе от первого лица («я», «мне», «меня», «мой»). Чтобы не выдавать пол, избегай форм прошедшего времени, раскрывающих его («сделал/сделала») — используй настоящее/будущее: «помогу», «подскажу», «передаю», «уточню». Исключение: если клиент прямо спросил «кто ты?» — ответь «я ассистент менеджера».

В `_SECURITY_RULES`: `(о себе — во множественном числе)` → `(о себе — в единственном числе, «я»)`.
Строка 65: «мы не видим содержимое вложения» → «я не вижу содержимое вложения».

**b) Динамический блок ПРИВЕТСТВИЕ** (новый, добавляется в `_build_system` по флагам `should_greet` / `is_first_message`)

- первое сообщение → «Поздоровайся один раз и спроси, как можно к клиенту обращаться (напр.: «Здравствуйте! Подскажите, как могу к вам обращаться?»). Больше в этом диалоге не здоровайся.»
- возврат после паузы >12ч → «Можешь поздороваться один раз. Имя повторно не спрашивай.»
- продолжение диалога (<12ч) → «НЕ здоровайся — вы уже общаетесь. Не пиши «здравствуйте»/«добрый день», сразу к сути.» ← чинит двойное приветствие при отправке тура.

**c) Контакт менеджера по контексту** (новое правило в `_BASE_RULES`)

> Если клиент сомневается, спрашивает про оплату/бронирование/контакты или ты завершаешь подбор — предложи оставить номер или позвонить менеджерам Наталье и Татьяне по +375 29 152 37 28 (в рабочее время 10:00–18:00). НЕ добавляй это приглашение в каждое сообщение — только когда уместно.

**d) Нет подходящих туров** (новое правило, точный текст заказчика с подстановкой направления + обязательный `===МЕНЕДЖЕР===`)

> «На данный момент, в нашем текущем ассортименте нет туров в {направление}.
> Я передаю ваш запрос нашим менеджерам Наталье и Татьяне — они смогут уточнить, есть ли возможность организации индивидуального тура или появятся ли такие направления в ближайшее время.
> Также вы можете позвонить им самостоятельно по телефону +375291523728 либо оставить свой контакт - девушки с радостью ответят на все вопросы в удобное для вас время.»

**e) Эскалация — текст клиенту + захват телефона** (`_ESCALATION_RULES`)

- при передаче менеджеру говорить клиенту, что передаёшь запрос менеджерам **Наталье и Татьяне**, и что он может сам позвонить по +375 29 152 37 28 в рабочее время;
- если клиент **оставил номер телефона** на любом этапе → обязательно `===МЕНЕДЖЕР===` с номером в поле «Телефон» + ответ «передал контакт менеджерам, с вами свяжутся в рабочее время»;
- расширить формат блока (опциональные поля):

```
===МЕНЕДЖЕР===
Причина: ...
Контекст: ...
Имя: ... (если клиент назвал)
Телефон: ... (если клиент оставил)
===МЕНЕДЖЕР===
```

**f) Сигнатуры** — `build_full_prompt(...)` и `_build_system(...)` получают доп. параметры `should_greet: bool = True`, `is_first_message: bool = True`.

### 2. `src/main.py`

**a) Логика приветствия** (в `process_with_ai`, после чтения сессии ~стр. 325)

- новый хелпер `_should_greet(prev_last_iso, is_first)`: `True`, если первое сообщение, нет `last_message_at`, или прошло ≥12ч (по аналогии с `is_manager_active` в `sessions.py:78`);
- `is_first = not history`; передать `should_greet` и `is_first` в `build_full_prompt`;
- перед `save_session` записать `session["last_message_at"] = datetime.now(timezone.utc).isoformat()` (также в `_process_non_text_safely`, чтобы таймер 12ч обновлялся).

**b) Парсинг эскалации → dict** (`_extract_escalation`, стр. 226)

- вернуть `{"reason", "context", "name", "phone"}` вместо кортежа; парсить также «Имя»/«Телефон»;
- обновить вызов (стр. 355–361) и проверку `not extracted`;
- в `notify_manager` (стр. 404) передавать `client_name`/`client_phone` из эскалации (с фолбэком на `booking_data`) — так оставленный номер попадёт менеджеру в Telegram.

**c) Самоидентификация** — в `_process_non_text_safely` (стр. 503) «Мы не видим содержимое вложения» → «Я не вижу содержимое вложения».

### 3. `src/db/sessions.py`

В `_new_session` (стр. 57) добавить `"last_message_at": None` для явности. Схема БД не меняется — поле живёт в JSON `state`.
