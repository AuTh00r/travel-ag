## 1. План

Файл: `docs/WORK-PLAN-instagram-non-text.md`

**Проблема:** `InstagramChannel.receive_message()` обрабатывал только `message.text`. Сообщения с вложениями/постами/ответами на истории без текста доходили до webhook, но не превращались в событие и терялись.

**Цель:** non-text сообщения создают `kind="user_non_text"`, передаются менеджеру через Telegram без вызова DeepSeek, клиент получает автоответ.

---

## 2. Реализация

### 2.1 `src/channels/instagram.py`

- **`_extract_non_text_metadata(message)`** — статический метод, детектирует `attachments`, `reply_to`, `referral`. Возвращает `{types, summary, has_text, text, raw_keys}` или пустой dict.
- **`receive_message()`** — теперь возвращает три типа событий: `kind="user"`, `kind="user_non_text"`, `kind="manager"`.
- **`_merge_sender_events()`** — мержит `user` и `user_non_text` события от одного sender в одном webhook POST. Meta присылает shared post и текст как два отдельных `messaging` entry.

### 2.2 `src/main.py`

- **`_process_non_text_safely()`** — обработчик без LLM:
  - Manager takeover gate
  - Telegram-уведомление менеджеру с типом и текстом клиента
  - Лимит 3 эскалации
  - Сохранение истории
  - Задержка 5s + проверка `_in_ai_processing` перед auto-ack (если AI уже обрабатывает текст — auto-ack не отправляется)
- **Webhook-ветка:**
  - `user_non_text` с текстом → AI (с аннотацией `[Клиент также отправил: ...]`)
  - `user_non_text` без текста → `_process_non_text_safely`
  - `_in_ai_processing` заполняется до создания задач (cross-request dedup)

### 2.3 `src/ai/prompts.py`

- Бот говорит о себе во множественном числе («мы», «нас», «нам»)
- Если прямо спросили «кто ты?» — «я ассистент менеджера»
- Смягчён тон для вложений: «Спасибо, мы не видим содержимое вложения, передали вопрос менеджеру»

### 2.4 Auto-ack текст

```
Спасибо! Мы не видим содержимое вложения, поэтому передали ваш вопрос менеджеру.
Он посмотрит и поможет 🙌
```

### 2.5 Telegram-уведомление

```
Клиент отправил не текстовое сообщение в Instagram.
Тип: вложение: image
Текст клиента: <текст или "без текста">
Бот не видит содержимое вложения/поста/истории.
```

|---|---|
| Non-text события терялись | `_extract_non_text_metadata` + `kind="user_non_text"` |

## 8. Сессия 3 — Telegram Markdown → plain text

### Проблема

Telegram-уведомления об эскалации падали с ошибкой `can't parse entities` на втором аккаунте (1218542962), хотя первый (904138085) получал то же сообщение. Markdown-парсер Telegram нестабилен — один текст принимает для одного chat_id, но не для другого.

### Решение

Убран `parse_mode: "Markdown"` — уведомления отправляются как plain text. Emoji и URL работают, форматирование (bold/italic) потеряно, но доставка гарантирована.

### Изменённые файлы

- `src/services/telegram_notify.py` — удалён `parse_mode`, удалена разметка `*текст*` и `` `текст` ``, удалена `_escape_md()`
- `tests/test_telegram_notify.py` — убран assert на `parse_mode`
